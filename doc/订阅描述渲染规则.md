# 订阅描述渲染规则

本文档用于说明 Admin 中“产品管理 / 订阅套餐”的 `description` 字段为什么在 User 端“套餐购买”页面没有按普通文字显示，以及管理员应当如何填写才能被用户端正确渲染。

## 0. 当前长期运营方案

当前长期运营方案已经从“一个 `description` 字段承载所有内容”调整为“卡片简介 + 结构化亮点 + 详细说明”三段式模型：

| 字段 | 用途 | 推荐填写方式 |
| --- | --- | --- |
| `short_description` | User 套餐卡片上的一句话简介 | 普通文本 |
| `features` | User 套餐卡片上的亮点列表 | Admin 可视化增删项，后端保存为 JSON 数组字符串 |
| `detail_format` | 详细说明格式 | `markdown` / `html` / `text` |
| `detail_content` | 详细说明正文 | Markdown 默认推荐，HTML 为高级模式 |
| `description` | 历史兼容字段 | 继续保留，给旧前端或旧数据兜底 |

结论先说：

- 新建/编辑订阅时，Admin 不再要求客户直接手写 JSON。
- User 套餐卡片优先读取 `short_description` 和 `features`。
- User 购买弹窗或独立购买页可渲染 `detail_content`，支持 Markdown、纯文本和安全 HTML。
- 旧 `description` JSON 仍然兼容；旧普通文本也会兜底显示，不再直接空白。

## 1. 当前数据链路

后台保存链路：

```text
Admin 产品管理
  -> 新建订阅 / 编辑订阅
  -> 内容页签
  -> 填写卡片简介、套餐亮点、详细说明
  -> 后端写入 proxy_subscribe.short_description / features / detail_format / detail_content
  -> 同时写入 legacy description 供旧链路兼容
```

用户展示链路：

```text
User 套餐购买页
  -> GET /v1/public/subscribe/catalog
  -> 返回 subscribe.short_description / features / detail_format / detail_content / description
  -> 优先按新字段渲染
  -> 新字段为空时回退解析 legacy description
```

关键边界：

- 后端 `CreateSubscribe` / `UpdateSubscribe` 接收新字段和旧 `description`，一起写入订阅模型。
- Public 订阅接口读取数据库中的新旧字段，一起放到返回值。
- User 前端优先读新字段；旧字段只作为兼容兜底。
- HTML 详情不会直接信任原始内容，User 端渲染前会做白名单净化。

所以当前问题的长期解决方式不是继续教客户手写 JSON，而是让 Admin 提供可视化编辑和 Markdown/HTML 详情编辑能力。

## 2. 用户端接受的 JSON 结构

旧 `description` 字段本身是一个字符串，但字符串内容必须是一段 JSON 对象。

完整结构：

```json
{
  "description": "套餐简介",
  "features": [
    {
      "type": "default",
      "icon": "uil:database",
      "label": "100 GB 流量"
    },
    {
      "type": "success",
      "icon": "uil:shield-check",
      "label": "支持高速节点"
    },
    {
      "type": "destructive",
      "icon": "uil:shield-exclamation",
      "label": "不支持独享 IP"
    }
  ]
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 用途 |
| --- | --- | --- | --- |
| `description` | string | 建议填写 | 套餐卡片顶部的一段简短说明 |
| `features` | array | 建议填写 | 套餐特性列表 |
| `features[].label` | string | 必填 | 单条特性的显示文案 |
| `features[].icon` | string | 可选 | Iconify 图标名，例如 `uil:database` |
| `features[].type` | string | 可选 | 控制颜色和状态，可用值见下文 |

`type` 可用值：

| type | 显示效果 | 适用场景 |
| --- | --- | --- |
| `default` | 默认文字和主色图标 | 普通能力说明 |
| `success` | 绿色图标 | 推荐、支持、已包含的能力 |
| `destructive` | 红色图标，文字置灰并加删除线 | 不支持、限制项、不可用能力 |

## 3. 后台应如何填写

在 Admin 的订阅表单中，“描述”是 JSON 编辑器，不是普通多行文本框。推荐直接填写对象结构：

```json
{
  "description": "适合轻度日常使用，包含基础节点和标准速度。",
  "features": [
    {
      "type": "success",
      "icon": "uil:database",
      "label": "100 GB 可用流量"
    },
    {
      "type": "default",
      "icon": "uil:server",
      "label": "基础节点可用"
    },
    {
      "type": "destructive",
      "icon": "uil:shield-exclamation",
      "label": "不含独享入口"
    }
  ]
}
```

保存到接口或数据库后，实际 `description` 字段会变成 JSON 字符串，例如：

```json
"{\"description\":\"适合轻度日常使用，包含基础节点和标准速度。\",\"features\":[{\"type\":\"success\",\"icon\":\"uil:database\",\"label\":\"100 GB 可用流量\"}]}"
```

这是正常现象。外层是 API/数据库字符串，内层才是用户端解析出来的 JSON 对象。

## 4. 为什么普通描述不显示

以下写法不会被当前用户端套餐卡片正常展示：

```text
适合轻度用户使用
```

历史原因：旧前端只做 `JSON.parse`，普通文本会解析失败。新前端已经增加普通文本兜底，但旧前端仍可能空白。

以下写法也不符合当前规则：

```json
[
  { "feature": "100 GB 流量", "support": true }
]
```

原因：这是其他客户端或其他面板可能使用的数组格式，但当前 NPanel User Web 购买页使用的是对象格式：`description + features`。

以下写法能解析，但不会显示有效内容：

```json
{
  "content": "适合轻度用户使用",
  "items": ["100 GB 流量"]
}
```

原因：字段名不匹配。当前页面只读取 `description` 和 `features`。

## 5. 页面具体渲染规则

当前 User 端有多个位置会使用同一套逻辑：

- 套餐购买页：`apps/user/src/sections/subscribe/index.tsx`
- 购买确认页：`apps/user/src/sections/purchasing/content.tsx`
- 首页产品展示：`apps/user/src/sections/main/product-showcase/content.tsx`

新渲染逻辑如下：

```text
优先读取 short_description
  -> 优先读取 features
  -> 新字段为空时解析 legacy description
  -> short_description 有值就显示一行简介
  -> features 逐条显示 icon + label
  -> type = success 时图标绿色
  -> type = destructive 时图标红色，文字置灰并删除线
  -> detail_content 按 detail_format 渲染到详情区域
  -> legacy description 解析失败时当普通文本兜底显示
```

注意：套餐卡片里的“商品详情 / 可用流量 / 连接速度 / 同时连接 IP 数”不是 `description` 渲染出来的，它来自套餐的独立字段：

- `traffic`
- `speed_limit`
- `device_limit`

截图里箭头指向的空白区域属于描述和特性展示区。如果 `description` 不是合法 JSON，或者 JSON 中 `description` 为空且 `features` 为空，这个区域就会看起来像没有显示内容。

## 7. 排查步骤

当客户反馈“Admin 填了描述，但 User 套餐购买页不显示”时，按以下顺序检查：

1. 打开 Admin 对应订阅，确认描述字段不是普通文本，而是合法 JSON 对象。
2. 检查 JSON 顶层是否包含 `description` 和 `features`。
3. 检查 `features` 是否是数组。
4. 检查每个 `features` 项是否至少有 `label`。
5. 检查 `type` 是否只使用 `default`、`success`、`destructive`。
6. 打开浏览器开发者工具或接口请求，检查 `/v1/public/subscribe/catalog` 返回的 `description` 是否仍是这段 JSON 字符串。
7. 如果接口返回正确但页面仍为空，检查前端构建是否已经部署到使用该规则的版本。

## 8. 历史数据迁移

后端提供了迁移脚本：

```bash
go run ./scripts/migrate_subscribe_descriptions.go --dsn 'user:pass@tcp(127.0.0.1:3306)/npanel?parseTime=true'
```

默认是 dry-run，只打印会迁移的订阅，不写数据库。

确认无误后再加 `--apply`：

```bash
go run ./scripts/migrate_subscribe_descriptions.go --dsn 'user:pass@tcp(127.0.0.1:3306)/npanel?parseTime=true' --apply
```

也可以通过环境变量传入 DSN：

```bash
NPANEL_DATABASE_DSN='user:pass@tcp(127.0.0.1:3306)/npanel?parseTime=true' go run ./scripts/migrate_subscribe_descriptions.go --apply
```

迁移规则：

- 旧对象格式 `{ "description": "...", "features": [...] }` 会拆到 `short_description` 和 `features`。
- 旧数组格式 `[{ "feature": "...", "support": true }]` 会转换成新的 `features` 数组。
- 旧普通文本会写入 `short_description`，并作为 `text` 格式写入 `detail_content`。
- 如果新字段已经有内容，脚本会跳过该订阅，避免覆盖人工编辑结果。

## 9. 维护建议

为了避免运营误填，Admin 应使用“内容”页签维护套餐说明：

- 卡片简介：写一句话，不要写长文。
- 套餐亮点：用可视化列表维护图标、文案和状态。
- 详细说明：默认使用 Markdown。
- HTML：只作为高级模式使用，适合确实需要自定义结构的运营页面。

HTML 不建议粘贴整页外部网页源码。User 端会移除脚本、事件属性、style 和非白名单标签，避免影响页面安全和布局。

如果未来需要兼容移动端或其他客户端的数组格式，例如：

```json
[
  { "feature": "100 GB 流量", "support": true }
]
```

需要单独在对应客户端增加兼容解析逻辑，不能只改后端保存字段。后端当前会同时返回新旧字段，客户端应优先使用新字段。
